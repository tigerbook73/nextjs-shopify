# /update-convention — 规范更新流程

当实现方案与现有规范冲突，或规范无法覆盖新场景时，执行以下引导流程。

## 步骤 1：识别冲突

询问用户（或从上下文中提取）：

1. **冲突的规范**：是 `docs/conventions/` 中的哪个文件、哪条规则？
2. **触发场景**：什么新的实现/需求导致了冲突？
3. **冲突类型**（帮助用户判断）：
   - A：实现方案选择不当 → 调整实现，不更新规范
   - B：规范未覆盖新场景，但方向正确 → 补充规范
   - C：引入新技术，旧规范不再适用 → 更新规范 + 创建 ADR
   - D：规范本身设计有误 → 修正规范 + 创建 ADR + 评估已有代码

**如果是 A 类冲突**：说明应该如何调整实现，流程结束。

## 步骤 2：起草规范变更

针对 B / C / D 类冲突，起草变更内容：

- 当前规范的相关条目（原文）
- 建议的新条目（具体文字）
- 影响范围：这个变更会影响哪些已有代码？

等待用户确认变更方向。

## 步骤 3：创建 ADR

针对 C / D 类冲突（涉及重要决策变更），创建 ADR：

1. 读取 `docs/adr/README.md` 获取当前最大编号
2. 创建新文件 `docs/adr/<下一个编号>-<kebab-case-标题>.md`
3. 按 ADR 模板填写：背景、决策、替代方案、后果
4. 更新 `docs/adr/README.md` 的决策索引表

## 步骤 4：更新规范文件

修改对应的 `docs/conventions/<规范文件>.md`：

- 精确修改相关条目，不做无关改动
- 如果新增了规则，同步更新 `docs/conventions/README.md` 的描述（如需要）

## 步骤 5：同步检查

- 若 `docs/conventions/architecture.md` 的「明确禁止事项」有变动，
  检查 `CLAUDE.md` 和 `AGENTS.md` 是否需要同步（通常不需要，它们只是引用）
- 若新规范影响 Copilot 补全，更新 `.github/copilot-instructions.md`

## 步骤 6：输出变更摘要

```
## 规范更新摘要

**冲突类型**：C / D 类
**修改的规范文件**：docs/conventions/xxx.md
**创建的 ADR**：docs/adr/XXXX-xxx.md
**变更摘要**：
- 旧规则：...
- 新规则：...
**影响范围**：...
**建议回溯检查的文件**：...（如有）
```
