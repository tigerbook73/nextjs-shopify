# /update-convention — 规范冲突引导更新流程

当实现方案与现有规范冲突，或规范无法覆盖新场景时，执行以下引导流程。

## 步骤 1：判断冲突类型

询问用户（或从上下文中提取）：

- 冲突的规范：是 `docs/conventions/` 中哪个文件、哪条规则？
- 触发场景：什么新实现/需求导致了冲突？
- 冲突类型：

| 类型 | 描述 | 处理方向 |
| ---- | ---- | -------- |
| A | 实现方案选择不当 | 调整实现，不更新规范 |
| B | 规范未覆盖新场景，但方向正确 | 补充规范条目 |
| C | 引入新技术，旧规范不再适用 | 更新规范 + 创建 ADR |
| D | 规范本身设计有误 | 修正规范 + 创建 ADR + 评估已有代码 |

**类型 A**：说明应如何调整实现，流程结束。

## 步骤 2：起草规范变更（B / C / D）

- 当前规范的相关条目（原文）
- 建议的新条目（具体文字）
- 影响范围：变更会影响哪些已有代码？

等待用户确认变更方向。

## 步骤 3：创建 ADR（C / D 类）

1. 读取 `docs/adr/README.md` 获取当前最大编号
2. 创建 `docs/adr/<下一个编号>-<kebab-case-标题>.md`
3. 按 ADR 模板填写：背景、决策、替代方案、后果
4. 更新 `docs/adr/README.md` 决策索引表

## 步骤 4：更新规范文件

精确修改对应的 `docs/conventions/<规范文件>.md`，不做无关改动。

## 步骤 5：输出变更摘要

```
## 规范更新摘要

冲突类型：<A/B/C/D>
修改的规范文件：docs/conventions/xxx.md
创建的 ADR：docs/adr/XXXX-xxx.md（如有）
变更摘要：
  旧规则：...
  新规则：...
影响范围：...
建议回溯检查的文件：...（如有）
```
