# AI 辅助开发工作流

## 工具选择矩阵

| 场景 | 推荐工具 | 原因 |
| ---- | -------- | ---- |
| 特性开发（新功能） | Claude Code | 支持 Plan Gate、规范感知、slash 命令 |
| 代码重构 | Claude Code | 需要多文件上下文和规划能力 |
| Bug 分析与修复 | Claude Code | 需要理解上下文和规范 |
| 快速技术问答 | ChatGPT / 搜索 | 不涉及项目代码的概念解答 |
| 代码自动补全 | GitHub Copilot | IDE 内嵌，适合行级补全 |

**原则**：涉及项目代码的实现决策，优先用 Claude Code，确保规范一致性。

## 标准开发流程

```
1. 任务接收
   └── 明确本次任务属于哪个 Phase，对应哪个功能

2. 规范检查（实现前）
   └── 阅读 docs/conventions/ 中相关规范文件
   └── 确认没有与现有规范冲突的实现方案

3. Plan Gate（满足触发条件时）
   └── 3+ 文件 / 影响用户可见行为 / 新功能 → 进入规划模式
   └── 等待用户确认后再实现

4. 功能文档（每个 Phase 必须，Plan Gate 批准后第一步）
   └── 在 docs/features/phase<N>-<name>/ 创建三件套文档
   └── REQUIREMENTS.md → DESIGN.md → PROGRESS.md
   └── 文档确认后再开始写代码，不得跳过或与代码同步生成

5. 实现
   └── 按规范编写代码
   └── 遇到规范冲突 → 停止，提出冲突，等待决策

6. 规范自检
   └── 运行 /check-conventions 审查改动
   └── 有偏离 → 修正代码 或 走 /update-convention 流程

7. 提交
   └── 格式：type(scope): description（Conventional Commits，英文）
   └── 较大功能更新 PROGRESS.md
```

## 规范冲突决策框架

当新实现与现有规范冲突时：

```
冲突是因为...

A. 实现方案选择不当
   → 调整实现，遵守现有规范

B. 规范未考虑到这个场景，但规范方向仍正确
   → 在规范中补充该场景的处理方式（无需创建 ADR）

C. 引入了新技术/框架，旧规范不再适用
   → 更新规范 + 创建 ADR 记录决策（用 /update-convention 引导）

D. 规范本身设计有误，需要根本性修正
   → 更新规范 + 创建 ADR + 评估已有代码是否需要回溯修正
```

## AI 生成代码审查清单

接受 AI 生成的代码前，逐条确认：

- [ ] **规范符合**：是否遵守 `docs/conventions/` 中的相关规范？
- [ ] **类型安全**：有无 `any` 类型？返回类型是否明确？
- [ ] **错误处理**：GraphQL 调用是否检查了 `userErrors`？Server Action 是否返回结构化结果？
- [ ] **分层正确**：API 调用是否在正确的层次（不在 Client Component 直接调 Shopify）？
- [ ] **无多余依赖**：是否引入了规范中禁止的包？

## 学习记录建议

每完成一个 Phase，写一段简短的回顾（可放在 PROGRESS.md 或单独笔记中）：

- 这个阶段用了什么 Next.js 能力？解决了什么问题？
- 遇到了哪些规范边界情况？如何处理的？
- Shopify 的哪个 API 对应了哪个业务场景？
- 有哪些 AI 生成的代码与预期不符？原因是什么？
